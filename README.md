### Создание unit-файла systemd

Файл `/etc/systemd/system/tasks.service`:

```ini
[Unit]
Description=Tasks Service
After=network.target

[Service]
Type=simple
User=tasksuser
WorkingDirectory=/opt/tasks
EnvironmentFile=/etc/tasks/tasks.env
ExecStart=/opt/tasks/tasks
Restart=always
RestartSec=2
NoNewPrivileges=true
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
```

## Объяснение ключевых параметров unit-файла

| Параметр | Значение | Назначение |
|----------|----------|------------|
| `User=tasksuser` | Запуск от непривилегированного пользователя | Безопасность |
| `EnvironmentFile` | Путь к файлу с переменными | Конфигурация отдельно от кода |
| `Restart=always` | Автоматический перезапуск при падении | Надёжность |
| `NoNewPrivileges=true` | Запрет получения новых привилегий | Дополнительная изоляция |
| `LimitNOFILE=65535` | Увеличение лимита открытых файлов | Производительность |

# Контрольные вопросы
1. **Что такое VPS и зачем он нужен backend-разработчику?**  
VPS (Virtual Private Server) — это виртуальный выделенный сервер, на котором можно разворачивать и запускать свои приложения в изолированной среде. Backend-разработчику VPS нужен для публикации сервисов в сети, чтобы они были доступны другим пользователям, для тестирования в реальных условиях и для отработки навыков администрирования, таких как настройка systemd, управление процессами и обновлениями.

2. **Почему запуск приложения на VPS отличается от локального запуска на компьютере разработчика?**  
На локальной машине приложение обычно запускается вручную из IDE или терминала и останавливается при завершении сессии. На VPS сервис должен работать постоянно, независимо от подключения по SSH, автоматически перезапускаться при сбоях и стартовать при загрузке системы. Кроме того, на VPS необходимо учитывать безопасность, права доступа, системные логи и управление через стандартные инструменты (systemd, journalctl).

3. **Для чего используется systemd?**  
systemd — это система инициализации и управления службами в современных Linux-дистрибутивах. Она позволяет запускать сервисы при старте ОС, автоматически перезапускать их в случае сбоя, контролировать состояние (запущен/остановлен), изолировать окружение и управлять процессами через единый интерфейс команд (systemctl). Без systemd приложение пришлось бы запускать вручную или через костыльные скрипты.

4. **Почему не рекомендуется запускать серверное приложение от root?**  
Запуск от root даёт приложению неограниченные права на всю систему. Ошибка или уязвимость в коде может привести к компрометации всего сервера, удалению критических файлов или доступу к чужим данным. Лучше создавать отдельного непривилегированного пользователя (например, tasksuser) и запускать сервис от его имени, ограничивая зону поражения.

5. **Зачем выносить конфигурацию в отдельный env-файл?**  
Вынос конфигурации (порты, адреса внешних сервисов, параметры подключения) в отдельный файл позволяет менять поведение приложения без перекомпиляции. Это упрощает развёртывание в разных средах (разработка, тестирование, продакшн) и не требует изменений в коде. Также это повышает безопасность: секреты (пароли, токены) не попадают в репозиторий и могут быть защищены правами доступа на сервере.

6. **Что делает параметр Restart=always?**  
Параметр `Restart=always` в unit-файле systemd указывает, что сервис должен автоматически перезапускаться при любом завершении его процесса, независимо от причины (ошибка, падение, завершение по сигналу). Это повышает надёжность: если приложение упадёт из-за временного сбоя, systemd поднимет его заново без вмешательства администратора.

7. **Для чего нужен EnvironmentFile в unit-файле?**  
`EnvironmentFile` указывает путь к файлу, из которого systemd загружает переменные окружения перед запуском сервиса. Это позволяет централизованно хранить конфигурацию приложения (например, `TASKS_PORT=8082`) в отдельном защищённом файле, не прописывая её прямо в юните. Так легче обновлять настройки без правки unit-файла и без перечитывания конфигурации systemd.

8. **Как проверить состояние службы через systemctl?**  
Команда `systemctl status tasks` показывает текущее состояние службы (active/inactive), время работы, основной процесс, последние логи и информацию о том, включён ли автозапуск. Также можно использовать `systemctl is-active tasks` для программной проверки (вернёт active/inactive). Это основной инструмент для быстрой диагностики.

9. **Как посмотреть логи сервиса через journalctl?**  
`journalctl -u tasks` выводит все логи, собранные systemd для данного сервиса. Полезные ключи: `-f` (следить за логами в реальном времени), `--no-pager` (без постраничного вывода), `-n 100` (последние 100 строк), `--since "1 hour ago"` (с определённого времени). Это заменяет чтение лог-файлов вручную.

10. **Что нужно сделать перед обновлением unit-файла systemd?**  
Перед изменением unit-файла (например, правкой `tasks.service`) необходимо выполнить `systemctl daemon-reload`, чтобы systemd перечитал конфигурацию. После этого можно перезапустить сервис (`systemctl restart tasks`). Без reload systemd продолжит использовать старую версию unit-файла.

11. **Почему полезно иметь процедуру отката версии?**  
При обновлении приложения новая версия может содержать критические ошибки, несовместимости или неправильную конфигурацию. Процедура отката позволяет быстро вернуться к заведомо рабочей предыдущей версии без длительного поиска причины сбоя. Это минимизирует время простоя сервиса и является стандартной практикой эксплуатации.

12. **Зачем в реальных системах часто используют NGINX перед приложением?**  
NGINX выступает в роли reverse proxy и балансировщика нагрузки. Он умеет обрабатывать статику, ограничивать число соединений, сжимать ответы, завершать TLS (HTTPS) и распределять трафик между несколькими экземплярами приложения. Также NGINX скрывает внутренние порты (например, 8082) от внешнего мира, что повышает безопасность и упрощает развёртывание. Приложение слушает localhost, а NGINX проксирует запросы на него, что является стандартом для production-сред.
